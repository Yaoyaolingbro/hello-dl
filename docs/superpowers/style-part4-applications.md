# Part 4 现代 AI 应用写作规范

> 本文件与 `writing-style.md` 配合使用，**不重复通用规则，只写 Part 4 特有的要求**。

---

## 核心定位

读者是**想进入科研或工业界**的人。他们不需要从头推导每个公式，但需要：

1. 知道这个方向的来龙去脉（时间脉络）
2. 理解每篇代表性工作的贡献是什么
3. 知道工程落地时的实际问题

数学是背景知识，不是主角。工程细节和历史脉络才是。

Part 4 采用**科研科普**写法：先把研究问题讲成读者能观察到的矛盾，再介绍论文如何改变问题的解法。正文不能写成论文摘要的中文拼接，也不能用排行榜代替发展史。

---

## 论文选择：只保留必读主线

每个小节采用“精选主线”，通常选择 5 至 8 篇必读论文。论文数量服从叙事需要，不为凑数量加入边缘工作。

一篇工作满足以下至少一项时，才进入正文主线：

- 定义或重塑了任务范式
- 解决了上一代方法的关键瓶颈
- 成为后续研究反复复用的技术基础
- 代表当前仍有研究或工程影响力的路线

同一类方法有很多增量版本时，正文只讲最能说明技术转折的一篇。其余工作可以在一句话中交代，不展开模型清单。

每篇代表工作必须回答四个问题：

1. 当时大家卡在哪里？
2. 这篇论文改变了哪个假设或模块？
3. 实验说明它真正改善了什么？
4. 它留下了什么问题？

---

## 写作前的论文调研

动笔前先确认本节的任务边界和必读论文，再逐篇阅读原始材料。至少检查：

- 摘要和引言：作者认为之前的方法卡在哪里
- 方法部分：真正新增的机制是什么
- 实验和消融：哪些结论有数据支持
- 局限部分：哪些问题由作者明确承认
- 官方项目页或代码仓库：模型、数据和复现信息是否有更新

以下信息必须回到原论文或官方页面核验，不能依赖搜索摘要、二手博客或模型记忆：

- 论文名称、作者、年份和会议
- 数据集、指标、速度、参数量和训练成本
- “首次提出”“第一个做到”等优先权表述
- 作者明确陈述的局限
- 截至写作日期仍成立的趋势和开放问题

遇到互相矛盾的数字时，先检查输入分辨率、训练数据、预训练方式、评测协议和硬件。比较条件不一致，就不要把数字并排写成优劣结论。

## Paper 与 Project 链接

章节开头的 `!!! info "参考资料"` 集中列出本节的必读论文。正文第一次提到论文时，仍然提供链接：

```markdown
NeRF（[Paper](https://arxiv.org/abs/2003.08934) | [Project](https://www.matthewtancik.com/nerf)）
```

链接来源按以下顺序选择：

1. 会议或期刊正式页面、OpenReview、作者提供的 arXiv 页面
2. 作者或研究机构维护的官方项目页
3. 作者维护的官方代码仓库

有官方项目页时，`Project` 指向项目页；没有项目页时可以指向官方代码仓库；两者都没有时只写 `Paper`。不要用 Papers With Code、论文聚合转载页或非官方复现仓库代替官方链接。

正文只在第一次出现时附链接。后文再次提到同一工作，直接使用论文简称。

## 章节结构

```
!!! info "参考资料"

## 直觉 (Intuition)          ← 任务是什么，为什么难，核心方法是什么

## 任务定义                  ← 输入输出、任务边界、常用指标

## 发展脉络                  ← 时间线，关键节点的工作和贡献

## 核心方法                  ← 每类方法的原理（数学点到为止，指向 Part 3 深入）

## 工程实践                  ← 落地时遇到的真实问题

## 开放问题                  ← 这个方向现在还有什么没有解决好
```

## 时间脉络叙事（最重要）

Part 4 的叙事主线是**时间**，不是逻辑推导。读者需要理解：当某篇论文出现时，它在解决什么当时的痛点。

**框架模板：**

> **背景**：2014 年，目标检测还在用 R-CNN，每张图要跑 2000 次 CNN，推理需要 49 秒。
>
> **问题**：这在任何实时应用里都不可用。
>
> **突破**：YOLO（2016）把检测变成一次 forward pass，速度从 49 秒降到 45fps。代价是小目标漏检率偏高。
>
> **后续**：SSD 和 YOLOv2 各自在不同方向做了改进……

每个代表性工作要说清楚：**它出现时解决了什么问题，引入了什么新问题，后来怎么被解决的。**

时间只是排序工具，研究问题才是段落之间的因果关系。不要写成“某年有 A，下一年有 B”的论文年表。

## 价值陈述优先于原理陈述

不要用大量篇幅推导一个方法的数学细节，把那部分留给 Part 3 的链接。要花精力说清楚：

- 这个方法做到了什么（性能、速度、可用性）
- 相比之前的方法好在哪里（具体的提升）
- 它的局限是什么（为什么后面的工作还要继续）

> NeRF（2020）第一次用神经网络实现了照片级的新视角合成，在 LLFF 数据集上 PSNR 提升约 5dB。代价是单场景训练需要 1-2 天，无法泛化到新场景。这两个局限直接催生了后来的 instant-ngp 和 zero-shot 重建方向。

如果数字不是理解技术转折所必需的，就优先解释实验现象。例如，“在遮挡区域仍能保持边界”通常比单独抄一个平均分更能帮助读者理解方法。

## 工程实践的权重

Part 4 的 `!!! tip "面试 / 工程重点"` 密度可以比其他 Part 更高（每节 2-5 个），但每个必须是真实工程中会遇到的问题：

**好的工程 tip（真实遇到的）：**
```markdown
!!! tip "工程重点"
    NeRF 在实际部署时最常见的问题是场景边界处理：训练集里没有覆盖到的视角，
    渲染结果会有明显的浮影。解决方法是在训练时加入背景建模（如 NeRF++），
    或者用 mip-NeRF 的 cone casting 缓解。
```

**差的工程 tip（过于显然）：**
```markdown
!!! tip "工程重点"
    使用 GPU 可以加速训练。  ← 这不是工程 tip，这是废话
```

面试类 tip 侧重于**"面试官真的会问、答不上来会扣分"的问题**：

```markdown
!!! tip "面试高频"
    DETR 为什么收敛慢？
    
    原因是 bipartite matching 的优化目标不稳定——同一个 GT box 在不同训练步里
    可能被分配给不同的 query，梯度信号相互干扰。后来的 DN-DETR 通过加入
    去噪训练解决了这个问题，收敛从 500 epoch 降到 50 epoch。
```

## 数学的使用原则

Part 4 的数学以**公式直觉**为主，不推导：

- 给出核心公式，解释每项含义，不展开推导
- 如果某个方法的数学在 Part 3 已经讲过，直接链接，不重复
- 复杂的数学证明用"可以证明……"带过，给出参考文献

```markdown
体渲染方程描述光线沿方向 $\mathbf{d}$ 穿过场景时的颜色积分：

$$C(\mathbf{r}) = \int_{t_n}^{t_f} T(t) \, \sigma(\mathbf{r}(t)) \, \mathbf{c}(\mathbf{r}(t), \mathbf{d}) \, dt$$

其中 $T(t)$ 是透射率（光线没被遮挡的概率），$\sigma$ 是体密度，$\mathbf{c}$ 是颜色。
推导见 Max 1995 年的原始体渲染论文。
```

## Benchmark 数字的使用

引用性能数字时：

- 说清楚在什么数据集上，什么指标
- 给出绝对值（不只是"提升了 X%"）
- 说明是否是当时的 SOTA，以及现在还是不是
- 写清楚会影响公平比较的训练数据、输入尺寸或预训练条件

> 在 ImageNet 上，ViT-L/16 达到 87.1% Top-1（2021 年时 SOTA），但需要 300M 参数和大量预训练数据。同期的 EfficientNet-B7 用 66M 参数达到 84.3%，在数据受限场景下更实用。

不要随意引用没有说明来源的数字。

## 开放问题必写

每节结尾说清楚**这个方向现在还在哪里卡着**，这对科研读者最有价值。标题下必须注明资料核验时间，例如：

> 以下判断基于截至 2026 年 6 月公开的论文与项目资料。

> 3DGS 的主要开放问题：（1）内存占用随场景复杂度线性增长，大场景（城市级别）还没有好的解决方案；（2）动态场景下的高斯追踪还不够稳定；（3）和语义理解的结合还处于早期阶段。

开放问题必须能落到可研究的矛盾上，例如数据、泛化、效率、可靠性或评测，而不是笼统地说“效果还可以继续提升”。如果问题已经被后续工作大幅缓解，要写清楚剩余边界，不能沿用旧论文的结论冒充现状。

## 交稿前额外检查

- [ ] 叙事是按时间脉络展开的吗？每个代表性工作都说了"它出现时解决了什么问题"吗？
- [ ] 正文主线是否只保留了真正必要的论文？
- [ ] 每篇主线论文首次出现时是否提供了官方 Paper 链接，以及存在时的 Project 链接？
- [ ] 论文名称、年份、会议和关键数字是否回到原始来源核验？
- [ ] 有没有说清楚每个方法的局限（以及后来怎么被改进的）？
- [ ] 工程 tip 是真实工程问题吗（不是显然的废话）？
- [ ] 引用性能数字时说明了数据集和指标吗？
- [ ] 节末有注明核验时间的"开放问题"或"这个方向现在的挑战"吗？
- [ ] 数学有没有超出必要范围，可以用链接替代的推导是否已经替换了？
