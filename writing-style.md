# Writing Style Guide

> 每次开始编写新章节前，必须先读这个文件。

---

## 核心理念

这本书的读者是刚入门 AI 的同学，他们有大学数学基础，但没在科研中用过。
**目标不仅是严谨，而是让人读懂**。一个概念如果讲不清楚，往往是因为我们自己还没真正理解它。
数学内容要深入浅出，不能不严谨，但不要讲的过于复杂。

写作的标准只有一个：**你的室友能听懂吗？**

当内容是涉及应用层面的，一定要讲清楚：任务 → 为什么难 → 需要什么能力 → 对应模型/方法 → 现在还有什么没有被解决好。

并且，如果有面试相关的工程上的技术细节，我建议你也进行相应的补充。通过admonition的形式进行点缀，**但不要每段都有**，要真正有干货的才写。

---

## 声音与语气

**要做到：**
- 像给朋友解释一样写。用"我们"而不是"读者"。
- 一句话只说一件事。
- 先说结论，再解释原因。必要的时候可以举例。
- 大白话优先，能用"变大"就不用"单调递增"。
- 术语第一次出现时，用括号给出英文原文，例如：梯度下降 (Gradient Descent)。

**不要做：**
- 不要铺垫太长才进入正题。
- 不要把定义当解释。"熵是信息量的期望"不是解释，是重复定义。
- 不要一个段落里塞超过一个新概念。
- 不要写"由上文可知"、"显然"、"不难证明"这类话。
- 不要为了"严谨"而加大量限定词，把简单的事说复杂。
- 不要随便出现新的名词，如若必须出现，请在下方添加说明，立刻解释它是什么。

## 每章必有的开场：直觉段

每章正文的第一部分固定是"直觉"，格式如下，**不超过 5 句话**：

```
## 直觉 (Intuition)

[这个东西在解决什么问题？]
[输入是什么，输出是什么？]
[核心思路一句话是什么？]
[可选：和读者已知的东西类比]
```

示例（梯度下降）：
> 我们有一个损失函数，想找到让它最小的参数。
> 输入是当前参数，输出是更新后的参数。
> 核心思路是：沿着"下坡方向"走一小步。
> 就像蒙着眼睛下山，每次感受脚下哪个方向更陡，就往那走。

## Markdown 语法注意
由于mkdocs的渲染问题，有一下一些不同于寻常markdown写法的规范：

- 编写有序和无序列表时，前后必须有至少两个空行。
- 需要换行的地方，必须有两个换行，即中间空出来一行，才能够正确渲染。

## 数学的写法

**规则一：公式之前必须有一句话说它在讲什么。**

不好的写法：
> 定义损失函数 $\mathcal{L} = -\sum_i y_i \log \hat{y}_i$。

好的写法：
> 我们希望预测概率 $\hat{y}_i$ 尽量接近真实标签 $y_i$，交叉熵损失就是在量化这个差距：
> $$\mathcal{L} = -\sum_i y_i \log \hat{y}_i$$

**规则二：每个新符号第一次出现时，立即解释含义。**

> 其中 $\mathbf{W} \in \mathbb{R}^{d_{\text{out}} \times d_{\text{in}}}$ 是权重矩阵，$d_{\text{in}}$ 是输入维度。

**规则三：符号严格对齐原论文。**

章节开头的 `!!! info` 块中列出主要参考论文，之后所有符号与该论文保持一致。
如果多篇论文符号冲突，在 `!!! note` 块中显式说明本章的统一约定。

**规则四：复杂推导分步写，每步说目的。**

```
第一步：展开平方项，目的是把交叉项分离出来
第二步：对 θ 求导，令导数为零
第三步：整理得到解析解
```

## 代码的写法

**Part 1 数学基础章节不写代码**，改用手算数值例子和示意图（见下方"图的使用规范"）。

Part 2 及以上章节才写代码，规则如下：

- 只写核心逻辑，辅助代码用 `# ...` 省略。
- 注释说**为什么**，不说是什么（变量名已经说了是什么）。
- 每段代码前一句话说明它在验证或实现什么。
- 输出结果写在注释里，读者不用跑就能知道结果。

```python
# 验证 softmax 数值稳定性
z = torch.tensor([1000.0, 1001.0, 1002.0])
probs = F.softmax(z, dim=0)  # 不会溢出，因为 PyTorch 内部做了减最大值
# tensor([0.0900, 0.2447, 0.6652])
```

代码语言：Python + PyTorch，保持最新稳定版写法。

## Admonition 使用规范

只用以下四种，不要滥用：

```markdown
!!! info "参考资料"
    章节开头，列出论文和优质资源。

!!! note "直觉小结"
    某段推导结束后，用一句话总结直觉含义。

!!! tip "面试 / 工程重点"
    工程落地细节、面试高频考点，点缀使用，不要每段都有。

!!! warning "常见误区"
    真实存在的常见错误，不要为了写而写。
```

`pymdownx.tabbed` 仅用于真正需要**并排对比**的场合：两种等价实现、不同变体的对比。不要把正文内容拆进 tab。

## index.md 规范

每个目录下的 `index.md` 是该章节的**导读页**，固定结构如下：

```markdown
# 章节标题

一句话说明本章解决什么问题、覆盖什么范围。

## 你将学到

| 小节 | 核心内容 | 前置依赖 |
|------|----------|----------|
| [小节名](file.md) | 一句话描述 | 无 / 链接到前置章节 |

## 本章知识地图

（可选）一个 Mermaid 图，展示本章各小节的依赖/顺序关系。
```

原则：
- 导读页**不写正文内容**，只做索引和导航。
- "你将学到"表格让读者在开始前知道投入和收获。
- "前置依赖"帮助读者决定是否需要先回顾其他章节。

## 每章编写前的 Checklist

- [ ] 重读本文件（writing-style.md）
- [ ] 搜索该主题的优质 blog / 讲解视频（搜不到也没关系，继续写）
- [ ] 确认主要参考论文，记录符号约定
- [ ] 写直觉段（≤5 句话）
- [ ] 逐节穿插写：文字 → 公式 → 代码，不要分离
- [ ] 检查：每个新符号是否有解释？每个公式前是否有一句话说明含义？
- [ ] 在 reference.md 中补充本章用到的新资源

## 图的使用规范

- **结构图 / 流程图**：用 Mermaid，保持简洁，不超过 10 个节点。
- **直觉性示意图**：主动从网络上找高质量图片嵌入，图片来源优先级：
  1. **Wikipedia Commons**（最稳定，永久 URL，CC 协议可直接使用）：在 commons.wikimedia.org 搜索概念名，使用页面"File"链接下的直接图片 URL
  2. **论文原图**：用 arxiv 或 semantic scholar 上论文页的图片直链，注明"来源：论文名"
  3. **教程截图**（3Blue1Brown、distill.pub 等）：只引用 URL，不转存，标注出处
- 图片都可以点击放大（glightbox 已配置）。
- 每张图下方写一句 caption 说明图在展示什么，格式：`*[描述内容]。来源：[超链接]*`
- **Part 1 数学基础每小节至少插入一张示意图**，重点覆盖：几何直觉（向量、变换、分解）、函数图像（梯度方向场）、算法流程。

### 嵌入图片的 Markdown 格式

```markdown
![特征向量在矩阵变换下方向不变](图片直接URL)

*特征向量（蓝色、绿色）在矩阵变换后方向不变，只被缩放；其他向量（灰色）方向改变。来源：[Wikipedia](https://en.wikipedia.org/wiki/Eigenvalues_and_eigenvectors)*
```

搜索建议：搜索词用英文更容易找到高质量数学可视化，如 `eigenvalue eigenvector visualization wikipedia`、`SVD singular value decomposition geometry diagram`。

## 语言规范

- 中文行文，英文术语第一次出现括号标注。
- 之后同一术语可以直接用中文或英文，保持全文一致。
- 数字：10 以内用汉字，10 以上用阿拉伯数字。
- 标点：中文句子用中文标点，公式内用英文标点。

## 去除 AI 写作痕迹

> 技术内容写得再严谨，如果读起来像机器生成的，读者就会失去信任感。
> 下面这些是最容易暴露 AI 痕迹的模式，每次写完都要对照检查。

### 五条核心原则

1. **删填充词** — "此外"、"值得注意的是"、"不难看出"、"总的来说" 直接删掉
2. **打破对称结构** — 避免"不仅……而且……"、"一方面……另一方面……"这类公式
3. **变化句子节奏** — 长句和短句交替。连续三句长度相同，打断其中一句
4. **信任读者** — 不解释隐喻，不重复刚说过的结论，不加"综上所述"
5. **删金句** — 听起来像可以截图转发的句子，重写

### 禁用词汇

以下词汇在正文中一律不用（`!!! tip` 等 admonition 内同样适用）：

| 禁用 | 改用 |
|------|------|
| 此外、另外、值得注意的是 | 直接开始下一句 |
| 不仅……而且…… | 直接说第二点 |
| 充满活力的、令人叹为观止的、开创性的 | 具体描述它做了什么 |
| 专家认为、研究表明、有观点认为 | 给出具体来源，或删去 |
| 标志着、见证了、是……的证明 | 直接说事实 |
| 在……的背景下、随着……的不断发展 | 删去，直接进主题 |
| 复杂性、格局、织锦（比喻用法） | 换成具体的词 |
| 综上所述、由此可见 | 删去 |

### 结构禁忌

- **三段式列举**：尽量用两项或四项，三项看起来最像 AI
- **破折号戏剧化**：`——` 不用来"揭晓答案"或制造停顿感
- **粗体滥用**：一段话里最多一处粗体，用于真正需要强调的术语
- **内联标题列表**：`**名词：** 解释` 这种格式改成正常散文或普通列表

### 注入真实感

技术写作也可以有态度：

- 遇到反直觉的结论，说"这有点奇怪"，不要假装显然
- 遇到工程上常踩的坑，说"我第一次用的时候也在这里卡了很久"
- 公式推导中间，可以说"先别管这一步怎么来的，记住结论"
- 不确定的地方，说"这里的直觉我还没完全想清楚"，比假装权威好

### 交稿前快速检查

- [ ] 有没有连续三句句子长度一样？
- [ ] 有没有段落以短小精悍的"金句"结尾？（连续两段这样就要改一个）
- [ ] 有没有"不仅……而且……"或三段式列举？
- [ ] 有没有用了禁用词汇表里的词？
- [ ] 大声读一遍，有没有哪句话听起来不像人说话？
